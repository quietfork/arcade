## 1. プロジェクト概要

### 1.1 プロダクトの位置づけ
複数のローカルプロジェクトに対して Claude Code セッションを起動・並列管理するための、デスクトップ専用ランチャーアプリケーション。Apache Superset のダッシュボードに着想を得たグリッドレイアウト UI を採用し、ターミナルペインを縦横自由にリサイズ・配置できることを特徴とする。

### 1.2 背景
- Claude Code を複数プロジェクトで並行運用する際、`cd <path> && claude --dangerously-skip-permissions` を毎回手動実行する手間がある。
- tmux / Windows Terminal / VS Code 統合ターミナルでは、**プロジェクトカタログ + ワンクリック起動 + 自由なグリッド配置**を同時に満たす UI がない。
- Shogun Framework など複数エージェント並列実行のワークフローと相性の良い、Claude Code 専用ハブが必要。
- ブラウザ上ではなくローカルネイティブで動作し、起動・終了が軽快であることが要件。

### 1.3 目的・ゴール
- 登録済みプロジェクトの起動を**1クリック**で完了させる。
- 複数の Claude Code セッションを**視覚的に並列管理**できる UI を提供する。
- ペインの自由なリサイズ・配置により、マルチタスク時の認知負荷を低減する。

### 1.4 スコープ

#### In Scope (v1)
- プロジェクト(ローカルディレクトリ)の登録・編集・削除・一覧表示
- 登録プロジェクトからの Claude Code セッション起動(`--dangerously-skip-permissions` 自動付与)
- 複数ターミナルペインのグリッドレイアウト表示
- ペインのドラッグによる移動、縦横リサイズ、追加、閉じる
- レイアウト・プロジェクトカタログのローカル永続化

#### Out of Scope (v1)
- リモートホスト(SSH 越し)での Claude Code 起動
- Claude Code 以外の任意コマンド実行(v2 以降で検討)
- セッションログの永続化・検索
- チーム共有・クラウド同期
- プラグイン機構

## 3. システム構成

### 3.1 技術スタック

| レイヤ | 採用技術 | 採用理由 |
|---|---|---|
| アプリフレームワーク | Wails v2 | Go + WebView でバイナリが軽量。既存資産・経験あり。 |
| バックエンド言語 | Go 1.22+ | Wails 標準。PTY 制御も成熟。 |
| PTY ライブラリ | `github.com/aymanbagabas/go-pty` | Windows ConPTY / POSIX PTY をクロスプラットフォーム対応。 |
| フロントエンドフレームワーク | React 18 + TypeScript | Wails テンプレート標準。 |
| グリッドレイアウト | `react-grid-layout` | Superset 採用実績。縦横ドラッグリサイズ、レイアウト JSON 化が容易。 |
| ターミナル描画 | `@xterm/xterm` + `@xterm/addon-fit` | デファクト。VS Code 内部でも使用。 |
| 永続化 | JSON ファイル | RDB 不要、編集容易、バックアップ・移行が単純。 |

## 5. 非機能要件

### 5.1 性能要件
- アプリ起動から操作可能になるまで **3 秒以内**(SSD・通常スペック PC)。
- ペイン追加(クリックから描画完了まで)**1 秒以内**。
- 同時起動可能なペイン数: **最大 12 個**(性能保証ライン)。それ以上はベストエフォート。
- xterm の描画 FPS: 連続出力時に **30 FPS 以上**を維持。

### 5.2 互換性要件
- OS: Windows 10/11(x64) を **第一ターゲット**。
- 将来的に macOS / Linux にも対応可能であること(Wails と go-pty の特性により実現可能)。
- Claude Code バージョン: 最新安定版に追従。`claude` 実行ファイルが PATH 上にあること。

### 5.3 信頼性要件
- アプリクラッシュ時、子プロセス(claude)もクリーンに終了させる(ゾンビ防止)。
- 永続化 JSON は書き込み時に一時ファイル経由 + rename によるアトミック更新を行う。
- レイアウト破損時は自動でデフォルトに戻し、破損ファイルを `.bak` で退避する。

### 5.4 セキュリティ要件
- `--dangerously-skip-permissions` を既定で付与するため、**起動対象が登録済みプロジェクトに限定される**ことを UI 上明示する。
- プロジェクト追加時、登録パスを正規化し、シンボリックリンクの解決結果も保存する。
- 設定ファイルはユーザのホームディレクトリ配下に格納し、他ユーザから読み取れない権限とする(Unix: 0600 / Windows ACL)。
- xterm の入力にコマンドインジェクションのリスクとなるパススルーがないことを担保する(xterm から Go 側へは raw bytes 経由)。

### 5.5 ユーザビリティ要件
- マウス操作のみで全主要機能(登録・起動・配置・終了)が完結すること。
- 全ボタン・操作にツールチップを付与する。
- 操作の取消(Undo)は必須としない(削除系のみ確認ダイアログ)。

### 5.6 保守性要件
- フロントエンド・バックエンドのインターフェース(Wails bindings)は型定義を共通化する。
- ビジネスロジックは Go 側に集約し、フロントエンドは描画と入力ハンドリングに専念する。

## 8. 外部インターフェース

### 8.1 Claude Code との連携
- 標準入出力をそのまま PTY 経由でやり取りする。
- `claude` 実行ファイルが PATH 上に存在することを前提とする。未検出時は起動エラーをペイン上に表示する。
- バージョン依存の機能(slash command 等)には介入せず、純粋なターミナルとして振る舞う。

### 8.2 OS との連携
- フォルダ選択ダイアログは Wails の `OpenDirectoryDialog` を使用。
- ファイル書き込みはユーザーホームディレクトリ配下のみ。
- プロセス制御は Go 標準ライブラリ + go-pty。

## 11. リスクと対応

| リスク | 影響度 | 対応方針 |
|---|---|---|
| Windows ConPTY と Claude Code TUI の相性問題で表示崩れが発生する | 高 | Phase 0 で必ず実機検証。問題発生時は alacritty / Tabby のレンダリング設定を参考に修正。 |
| react-grid-layout のリサイズが xterm の fit と競合する | 中 | `ResizeObserver` でデバウンスし、`fitAddon.fit()` を呼ぶタイミングを制御。 |
| claude プロセスのゾンビ化 | 中 | アプリ終了時、登録セッションに対し SIGTERM → 待機 → SIGKILL の段階終了を実装。 |
| 多数同時起動時のメモリ消費 | 中 | スクロールバック上限と PTY バッファ上限を設定。同時起動数 12 を性能保証ラインとする。 |
| `--dangerously-skip-permissions` の常時付与によるユーザー誤操作 | 中 | 初回起動時に明示的な同意ダイアログを表示し、設定で解除可能にする。 |
| Claude Code のコマンド名・オプションが将来変更される | 低 | 起動コマンド・引数を設定可能にすることで将来変更に追従可能(FR-203)。 |

## 12. 受け入れ基準

MVP 完了の判断基準:

- [ ] 登録した 4 プロジェクトを順次クリックし、4 ペインに Claude Code が同時起動できること。
- [ ] 各ペインを縦横にドラッグリサイズしても TUI が崩れないこと。
- [ ] アプリを再起動しても、登録プロジェクトとレイアウトが復元されること。
